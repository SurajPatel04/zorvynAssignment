# Finance Data Processing and Access Control Backend

This is the backend implementation for the Finance Dashboard System, built to handle user roles, financial records, dynamic access control, and aggregated dashboard analytics.

## 🛠️ Technology Stack
- **Runtime & Framework**: Node.js, Express.js
- **Language**: TypeScript
- **Database**: MongoDB (via Mongoose)
- **Validation**: Zod (strict runtime schema parsing)
- **Authentication**: JWT (JSON Web Tokens) with HTTP-only cookies
- **Rate Limiting**: express-rate-limit (auth route protection)
- **API Documentation**: autogenerated via `swagger-autogen` and served via `swagger-ui-express`

---

## 📂 Project Structure

A clean, feature-based (modular) architecture was used to separate concerns:

```text
src/
├── config/           # Database, environment, and Swagger configurations
├── middlewares/      # Global error handling, Auth, Validation, and RBAC
├── modules/          # Feature/Domain modules (Business logic)
│   ├── auth/         # Login, Logout, Tokens routing & logic
│   ├── dashboard/    # Aggregation pipelines and analytics APIs
│   ├── permissions/  # Atomic system permission routing & logic
│   ├── roles/        # RBAC Role management
│   ├── transactions/ # Financial records CRUD
│   └── users/        # User management & status
├── scripts/          # Database seeders (Production limits & Dev mock data)
├── types/            # TypeScript declaration files (e.g., Express overrides)
├── utils/            # AsyncHandlers, custom ApiError, Response formatters, Rate Limiters
├── app.ts            # Express app setup and global routing
└── server.ts         # Application entry point & listener
```

## 🚀 Quick Setup & Emulating the Frontend Environment

### 1. Install Dependencies

```bash
npm install
```

### 2. Configure Environment Variables

Copy the `.env-sample` file and fill in your own values:

```bash
cp .env-sample .env
```

```dotenv
# Server
PORT=8000
NODE_ENV=development

# Database
MONGO_URI=mongodb+srv://<username>:<password>@<cluster>.mongodb.net/?appName=<appName>

# JWT
ACCESS_TOKEN_SECRET=<your_strong_random_secret>
ACCESS_TOKEN_EXPIRY=15m
REFRESH_TOKEN_SECRET=<your_strong_random_secret>
REFRESH_TOKEN_EXPIRY=10d

# Seed Admin Credentials (used by npm run seed)
ADMIN_EMAIL=admin@example.com
ADMIN_PASSWORD=Admin@1234

# Rate Limiting (optional — defaults are used if not set)
RATE_LIMIT_WINDOW=15m
RATE_LIMIT_MAX=10
```

> ⚠️ **Never commit your real `.env` file.** It is already listed in `.gitignore`.

### 3. Seed the Database

**⚠️ Important for Evaluators**: Please run both seed commands to populate roles, permissions, users, and transactions before starting the server.

```bash
# 1. Seed core system configuration (Permissions & Base Roles)
npm run seed

# 2. Seed test users and 150 fake transactions for dashboard data
npm run seed:dev
```

### 4. Start the Server

```bash
# Start in development mode (with hot-reloading)
npm run dev

# Or build and run for production
npm run build
npm start
```

## 🎯 How Functional Requirements Were Met

### 1. Dynamic Role-Based Access Control (RBAC)
Instead of hardcoding role limitations (`if (user.role === 'Admin')`), I designed a fully dynamic, database-driven RBAC system.
- The system has **Permissions** (e.g., `action: "read"`, `resource: "transactions"`).
- **Roles** are assigned an array of Permission references.
- **Users** are assigned a Role.
- The `authorize("action", "resource")` middleware intercepts the request, fetches the user's role and permissions fresh from the database on every request, and dynamically validates access. This ensures permission changes take effect immediately without requiring re-login.

### 2. Dashboard Aggregation Intelligence
To avoid fetching large arrays of data into Node.js memory just to calculate totals, I utilized **native MongoDB Aggregation Pipelines**:
- `/api/v1/dashboard/summary`: Uses `$group` to conditionally sum `$cond` income vs. expenses in a single query.
- `/api/v1/dashboard/trends`: Uses `$isoWeek` and `$month` to dynamically bucket transaction histories into time-series graphs based on the `?period=weekly|monthly` query.

### 3. Defenses, Search, Pagination & Soft Deletion
- **Soft Deletes**: Deleting a user or transaction sets `isDeleted: true` / `isActive: false` rather than destroying the document. All `find()` and `aggregate()` queries proactively enforce `{ isDeleted: false }`.
- **Search Support**: Regex-based text search is available on list endpoints via the `?search=` query parameter:
  - `GET /transactions?search=` — searches across `category` and `notes` fields.
  - `GET /users?search=` — searches across `fullName`, `email`, and `username` fields.
- **Lockout Prevention**: The `deactivateUser` controller contains specific safeguards preventing an Admin from deactivating their own account.
- **Pagination**: All list endpoints (`/transactions`, `/users`, `/roles`, `/permissions`) utilize parallel `Promise.all([Model.find().skip().limit(), Model.countDocuments()])` for highly performant pagination.

### 4. Rate Limiting
To protect unauthenticated endpoints from brute-force and abuse, `express-rate-limit` is applied on the most vulnerable auth routes:

| Endpoint | Limit |
|---|---|
| `POST /auth/login` | 10 requests / 15 min per IP |
| `POST /auth/refresh` | 10 requests / 15 min per IP |
| `POST /auth/register-user` | 10 requests / 15 min per IP |

The window duration and max attempts are configurable via environment variables (`RATE_LIMIT_WINDOW`, `RATE_LIMIT_MAX`), with sensible defaults so the app works out of the box without them.
